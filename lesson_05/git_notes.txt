Основні команди GIT

git init
  Ініціалізація нового git репозиторію

git status
  Виводить стан змін у файлах робочого каталогу та області підготовки

git add
  Додавання файлів до staging area (області підготовки)
    git add <filename> - додавання конкретного файла
    git add . - додавання всіх файлів

git rm
  Видалення файлів зі staging area (області підготовки)
    git rm --cached <filename>

git commit
  Створення нового коміту з файлами, які вже знаходяться в області підготовки, з додаванням короткого повідомлення, яке описує зроблені зміни
    git commit -m "commit message"

      git commit -a -m "commit message"
        Автоматичне додавання відслідковуваних файлів до staging area з їх миттєвим комітом

      git commit --amend --no-edit
        Зміна останнього коміту без зміни його повідомлення.
        Це корисно, коли ми хочемо додати забуті файли або виправити помилки в останньому коміті, але не хочемо змінювати вже написане коміт-повідомлення.

      git commit --amend -m "commit message"
        Зміна останнього коміту, але з заміною його коміт-повідомлення на нове.
        Дозволяє оновити останній коміт, якщо ми помилились в повідомленні коміту або хочемо його уточнити.

git reset
  Використовується для різних форм відкату змін у репозиторії

  git reset --soft HEAD^
    Переміщує останній коміт назад до області підготовки, але залишає робочий каталог недоторканим.
    Ми використовуємо це, коли хочемо зберегти зміни, зроблені в останньому коміті, та внести додаткові корективи перед новим комітом.

  git reset --hard HEAD
    Скидає всі незафіксовані зміни в робочому каталозі та області підготовки до останнього стану коміту, ефективно скасовуючи всі локальні зміни.

  git reset --hard HEAD~1
    Повертає на один коміт назад, видаляючи всі зміни, зроблені в останньому коміті, як у робочому каталозі, так і в області підготовки.

  git reset --hard HEAD~3
    Повертає нас на три коміти назад, видаляючи всі зміни, які були зроблені в останніх трьох комітах.

  Коротко:
    --soft
      ✅ прибирає коміт
      ✅ залишає зміни
      ✅ залишає staging

    --hard
      ❌ прибирає зміни
      ❌ прибирає staging
      ❌ прибирає все

  Мнемоніка:
    soft = “залиш код”
    hard = “спали все”

    забув файл у коміті?
      git reset --soft HEAD^
      git add forgotten_file
      git commit

    все зламав?
      git reset --hard HEAD


git log
  Використовується для перегляду історії комітів у репозиторії.

  git log
    Базова версія команди, яка показує детальну історію комітів, включаючи автора, дату та повідомлення коміту.

  git log --pretty=format:"%h - %an, %ar - %s"
    Кастомний вивід, показуючи історію комітів у спрощеному форматі. %h - це скорочений хеш коміту, %an - ім'я автора, %ar - відносна дата коміту, %s - повідомлення коміту.

  git log --since=2.weeks
    Виводить коміти, зроблені протягом останніх двох тижнів. Це корисно для перегляду змін за певний період.

  git log -p -2
    Показує останні два коміти разом з відмінностями (patch), внесеними у кожному коміті. Опція p дозволяє нам бачити конкретні зміни (додані чи видалені рядки) в кожному коміті.

git checkout
  Перемикання між різними версіями файлів або гілок у нашому репозиторії Git.

  git checkout -b <new-branch-name>
    Створює нову гілку і одразу перемикається на неї.

  git checkout main
    Перемикається на вказану гілку master.
    Якщо ми працювали на іншій гілці або переглядали старі коміти, ця команда повертає нас до останнього стану головної гілки.

  git switch main
    Більш актуальний варіант команди вище

  git checkout <commit-hash>
    Перехід до певного стану репозиторію, який відповідає вказаному хешу коміту. Це дозволяє нам переглянути код або відновити стан нашого проєкту на момент цього конкретного коміту.


git branch

  git branch
    Перелічує всі існуючі гілки в репозиторії, показуючи на якій гілці зараз знаходиться робоча копія.

  git branch <new-branch-name>
    Створює нову гілку з вказаною назвою.

  git branch -v
    Показує останні коміти в кожній гілці; це розширена версія команди git branch, яка включає додаткову інформацію про кожну гілку.

git merge <branch-name>
  Використовується для злиття змін з вказаної гілки <branch-name> до гілки, на якій зараз знаходяться. Це дозволяє інтегрувати роботу, завершену в одній гілці, з основною лінією розвитку, наприклад, злити функціонал з гілки розробки в головну гілку main.

git rebase
  Використовується для зміни бази (початкової точки) поточної гілки.
  Вона бере всі коміти в одній гілці і "перебазовує" їх на іншу гілку. Це часто використовується для підтримки лінійної історії змін, тобто замість створення коміту злиття, rebase переміщує коміти на верхівку гілки, на яку відбувається перебазування.

  Наприклад, якщо ми хочемо перебазувати нашу гілку feature на останні зміни з гілки main, ми використовуємо наступні команди:
    git checkout feature
    git rebase main


git remote
  git remote add origin <url>
    Використовується для додавання віддаленого репозиторію під назвою "origin" до нашого локального репозиторію. Ця URL-адреса вказує на місцезнаходження репозиторію на GitHub. Зазвичай "origin" є конвенційною назвою для основного віддаленого репозиторію.
    Після додавання цього віддаленого репозиторію, ми можемо пушити (відправляти) наші коміти та зміни до GitHub, а також фетчити (завантажувати) оновлення з нього.

  git remote
    Перелічення всіх віддалених репозиторіїв, які були додані до нашого локального репозиторію.

  git remote -v
    Виводить список віддалених репозиторіїв з URL-адресами для кожного псевдоніма (як для читання, так і для запису), даючи нам більш детальний огляд підключених віддалених сховищ.

  git remote rm origin
    Видаляє з локального репозиторію віддалене з'єднання під назвою "origin". Це корисно, якщо потрібно видалити посилання на старий або непотрібний віддалений репозиторій перед додаванням нового.

git push
  git push -u origin main
    Використовується для відправлення змін з локальної гілки main до віддаленого репозиторію, який має псевдонім origin.

  git push -u origin <branch-name>
    Використовується для встановлення віддаленої гілки на сервері, як основної для локальної гілки <branch-name>. Після виконання цієї команди, гілка <branch-name> буде асоційована з відповідною гілкою на сервері (origin).

    Параметр -u або --set-upstream означає, що для подальших операцій відправлення (push) та оновлення (pull) з цієї гілки на origin можна буде використовувати просто git push або git pull без додаткових аргументів.

git pull
  Використовується для оновлення нашої локальної гілки, витягуючи зміни з відповідної віддаленої гілки.
  Вона об'єднує в собі команди git fetch, яка завантажує зміни з віддаленого репозиторію, та git merge, яка об'єднує ці зміни з поточною гілкою.

  git pull --all
    Робить крок далі, оновлюючи всі локальні гілки, які мають відповідники на віддаленому сервері.
    Це корисно, коли працюється з багатьма гілками і потрібно забезпечити, що всі локальні гілки синхронізовані з їх віддаленими версіями.

git clone <url>
  Створення локальної копії вже існуючого репозиторію з вказаної URL-адреси. У цьому випадку, вона клонує репозиторій, розташований на GitHub, по заданому URL. Це включає всю історію комітів, гілки та файли проєкту.
  Ця команда корисна, коли потрібно почати працювати над існуючим проєктом або зробити копію проєкту для власного використання або внесення змін.

git stash
  Дозволяє тимчасово зберегти незафіксовані зміни у стек, щоб переключитися на інше завдання без втрати прогресу.

    git stash list — показує список всіх відкладених наборів змін.
    git stash push -m "comment" — зберігає зміни зі зручним описом.
    git stash pop — застосовує останній набір змін зі стеку та видаляє його.
    git stash pop stash@{2} — застосовує певний набір змін зі стеку (наприклад, третій).
    git stash apply — застосовує відкладений набір змін, але залишає його в стеку.
    git stash clear — очищає всі набори відкладених змін зі стеку.
